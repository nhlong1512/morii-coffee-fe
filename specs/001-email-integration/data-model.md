# Data Model: Email Integration for Welcome and Password Reset

**Feature**: Email Integration for Welcome and Password Reset
**Branch**: `001-email-integration`
**Date**: 2026-03-27

## Overview

This feature primarily involves UI enhancements and does not introduce new data entities. It works with existing authentication entities and adds client-side form state for password reset flows.

## Entities

### 1. Password Reset Form State (Client-Side Only)

**Purpose**: Temporary state for the reset password form

**Attributes**:
- `newPassword`: string - The new password entered by the user
- `confirmPassword`: string - Password confirmation field
- `isLoading`: boolean - Indicates form submission in progress
- `error`: string | null - Error message to display
- `isSuccess`: boolean - Indicates successful password reset

**Validation Rules**:
- `newPassword` and `confirmPassword` must match (client-side)
- `newPassword` must meet backend password complexity requirements:
  - At least 8 characters
  - Contains uppercase and lowercase letters
  - Contains numbers
  - Contains special characters

**State Transitions**:
```
Initial вҶ’ Loading (on submit)
Loading вҶ’ Success (on 200 OK from backend)
Loading вҶ’ Error (on 400/500 from backend or network failure)
Error вҶ’ Loading (on retry)
Success вҶ’ [auto-redirect to login after 2 seconds]
```

**Lifecycle**: Exists only during component mount, cleared on unmount

---

### 2. Forgot Password Form State (Client-Side Only)

**Purpose**: Temporary state for the forgot password request form

**Attributes**:
- `email`: string - Email address entered by user
- `isLoading`: boolean - Indicates form submission in progress
- `isSuccess`: boolean - Indicates request was sent (always true after submission)
- `error`: string | null - Client-side validation error (e.g., invalid email format)

**Validation Rules**:
- `email` must be valid email format (basic regex or HTML5 validation)
- No backend validation for email existence (security requirement)

**State Transitions**:
```
Initial вҶ’ Loading (on submit)
Loading вҶ’ Success (always, after backend responds)
```

**Important**: Never transitions to error state for "email not found" (security: prevent enumeration)

**Lifecycle**: Exists only during component mount

---

### 3. Reset Password URL Parameters (Read-Only)

**Purpose**: Query parameters passed from email link to reset password page

**Attributes**:
- `token`: string (Base64URL encoded) - Single-use password reset token from backend
- `email`: string (Base64URL encoded) - User's email address

**Validation Rules**:
- Both parameters must be present in URL
- Must remain encoded when sending to backend
- Optional decoding for display purposes only

**Source**: Email link generated by backend
```
http://localhost:3000/reset-password?token={encodedToken}&email={encodedEmail}
```

**Usage**:
```typescript
const searchParams = useSearchParams();
const token = searchParams.get("token");    // Keep encoded
const email = searchParams.get("email");    // Keep encoded

// Optional: Decode for display
const decodedEmail = base64UrlDecode(email);  // "user@example.com"
```

**Lifecycle**: Persists in URL until user navigates away

## Relationships

### Existing Entity: `ApiAuthResponse`

**Location**: `/src/types/api.ts`

**Structure**:
```typescript
interface ApiAuthResponse {
  accessToken: string;
  refreshToken: string;
  user: ApiUserProfile;
}
```

**Relationship**: No direct relationship with password reset forms. Password reset does NOT return auth tokens - user must log in after reset.

---

### Existing Entity: `ForgotPasswordRequest`

**Location**: `/src/interfaces/auth/index.ts`

**Structure**:
```typescript
interface ForgotPasswordRequest {
  email: string;  // Plain text (not encoded)
}
```

**Relationship**: Maps directly to forgot password form state

**API Endpoint**: `POST /api/v1/auth/forgot-password`

**Response**: Void (no data returned, always 200 OK)

---

### Existing Entity: `ResetPasswordRequest`

**Location**: `/src/interfaces/auth/index.ts`

**Structure**:
```typescript
interface ResetPasswordRequest {
  email: string;      // Base64URL encoded (from URL)
  token: string;      // Base64URL encoded (from URL)
  newPassword: string; // Plain text (from form)
}
```

**Relationship**: Maps to reset password form state + URL parameters

**API Endpoint**: `POST /api/v1/auth/reset-password`

**Response**: Void (no data returned)

**Success Response** (200):
```json
{
  "statusCode": 200,
  "message": "Success",
  "data": "Password reset successfully.",
  "errors": null
}
```

**Error Responses** (400):
```json
{
  "statusCode": 400,
  "message": "Invalid or expired reset token",
  "errors": ["Token has expired", "Token already used"]
}
```

## Validation Summary

### Client-Side Validation

| Field | Rule | Error Message |
|-------|------|---------------|
| Email (forgot password) | Valid email format | "Please enter a valid email address" |
| New Password | Not empty | (HTML required) |
| Confirm Password | Matches new password | "Passwords must match" |
| Token (URL) | Present in query params | "Invalid or missing reset link" |
| Email (URL) | Present in query params | "Invalid or missing reset link" |

### Backend Validation

| Field | Rule | Error Response |
|-------|------|----------------|
| Email (forgot password) | N/A (always returns 200) | N/A |
| Token | Not expired, not used | "Invalid or expired reset token" |
| Token | Valid format | "Invalid email or token format" |
| New Password | At least 8 chars | "Password does not meet requirements" |
| New Password | Contains upper+lower+number+special | "Password does not meet requirements" |

## Storage

### No Persistent Storage

This feature does NOT introduce any new persistent storage:
- No database tables
- No localStorage/sessionStorage (except existing auth tokens)
- No cookies (except existing i18n locale)
- All form state is ephemeral (React component state)

### Existing Storage (Used, Not Modified)

- **localStorage**: Auth tokens (via Zustand persist) - no changes
- **Cookie**: i18n locale preference - no changes

---

## Summary

The email integration feature works primarily with **transient form state** and **URL parameters**. It does not introduce new data entities or persistent storage. All API contracts and type definitions already exist in the codebase.

**Key Entities**:
1. Password Reset Form State (ephemeral)
2. Forgot Password Form State (ephemeral)
3. Reset Password URL Parameters (read-only from email link)

**Existing Entities** (reused):
- `ForgotPasswordRequest` (already defined)
- `ResetPasswordRequest` (already defined)
- `ApiAuthResponse` (not used by password reset)

**Validation**:
- Client-side: email format, password match
- Backend: token validity, password complexity

**Security**:
- No email enumeration (consistent responses)
- Single-use, time-limited tokens
- Base64URL encoding preserved until backend
